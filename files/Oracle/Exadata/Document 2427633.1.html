Purpose

Purpose of this document is to provide the quick overview of database under Exadata environment and also to explain the diagnostic and troubleshooting approach for database performance problems in Exadata environment.

Scope

 Scope of this document is limited ONLY to Database / SQL Performance in Exadata environment.

Details

Overview

Oracle Exadata Database Machine consists of Oracle Database servers, Exadata Storage Servers, and network components.
Oracle Exadata Database Machine is available as a full rack, half rack, quarter rack or eighth rack.
Oracle Exadata version starts from V1, V2, X2, X3, X4, X5, X6. X7 is the latest version of Exadata Database Machine.

Oracle Exadata Database Machine Full Rack consists of 8 Database servers (DB nodes) and 14 Exadata storage servers (CELL nodes).
Oracle Exadata Database Machine Half Rack consists of 4 Database servers (DB nodes) and 7 Exadata storage servers (CELL nodes).
Oracle Exadata Database Machine Quarter Rack consists of 2 Database servers (DB nodes) and 3 Exadata storage servers (CELL nodes).
Oracle Exadata Database Machine Eighth Rack consists of 2 Database servers (DB nodes) and 3 Exadata storage servers (CELL nodes).

NOTE: Eighth Rack has half the capacity or resources of quarter rack enabled at software level.

Each CELL nodes have FDD (Flash Drives) also in addition to general HDD (Hard disk). Each CELL node has 12 Hard disks.
The flask drives in each CELL can be used for enabling Smart Flash Cache.

Each DB nodes have 48 CPUs (a.k.a Cores) and each CELL nodes have 20 CPUs (a.k.a Cores). Refer the pictorial representation of Exadata Database Machine Full Rack below.

Troubleshooting Approach

Standard & Expected IOPS and Latency for Cell IO

Maximum IOPS(read+write) for CELL hardDisk is 200 (IOPS per disk). Anything above 150 per disk is of concern.
Since there are 12 disks in each CELL, the maximum IOPS for each CELL(HardDisks) is 2500.

Maximum IOPS for one FlashDisk in a CELL is 15K and for aggregate FlashDisks in a CELL is 100,000 IOPS.

Expected IO latency for CELL HDD(Hard Disk) is around 10-15 ms and for FDD(Flash Disk) is around 2 ms.

 

Check the below sections from AWR report

Check the Exadata Server Health for any Warning or Critical alerts.

Check the Exadata Statistics for Outliers, Server & Smart Statistics.
Outliers are cells or disks whose average performance is outside the normal range.

Check the Cell Server Statistics for IOPS and IO Latency

Check the IOPS for FDD and HDD whether it is in acceptable range

We can see above the Average IOPS for Flash Disk is around 800 and for Hard Disk is around 80 which is very good.
Note: The optimal IOPS for single HDD is around 150 and for single FDD is 15K. However, we need to compare it with baselines to report an IO problem.

Check for Smart IO statistics to identify whether Smart Scan is happened or not

We can see that there is around 84% efficiency for Cell Offload which means smart scan is happening.
Similarly, we can see the storage index is used and reduced around 12% of I/Os. 

Improving IO Performance using Smart Flash Cache

Exadata Storage Server provides Exadata smart flash cache, a caching mechanism for frequently accessed data. Using Exadata smart flash cache, an Exadata cell can support approximately around 100,000 IOPS.

Smart Flash Cache can be created from flash disk in the cell nodes.

From Exadata Storage Server release 11.2.3.2.0, Exadata Smart Flash Cache can operate either as a Write-through cache or a Write-back cache. Write-through mode is best suited for random repeated reads commonly found in OLTP applications. Write-back mode is best suited for write-intensive applications.

Write-Through -> first write to physical disks and then to flash cache for faster reads.
Write-Back -> first write to flash cache and then to physical disk for both faster reads and writes.

Refer Document 1500257.1 Exadata Write-Back Flash Cache - FAQ for more details.

Using Exadata Smart Flash Log

Exadata Smart Flash Log provides a mechanism for improving the latency of database redo log write operations.
Using Exadata Smart Flash Log, redo records are written simultaneously to the disk-based online redolog files and Exadata Smart Flash Log.
Exadata Storage Server software waits for the faster write operation, whether it is to hard disk or Exadata Smart Flash Log.
When the hard disks containing the online redolog files experience slow IO due to heavy load, the write operations to Exadata Smart Flash Log will typically complete faster and provide consistently good response times. Log file sync waits due to Slow IO can be completely alleviated using Smart Flash Log.

Refer Document 1372894.1 Exadata Smart Flash Logging FAQ for more details.

Exadata Smart Scan

One of the most powerful features of Database Machine is that data search and retrieval processing can be offloaded to Exadata Storage Servers.
This feature is popularly known as Smart Scan or Cell Offloading. Using this feature, Oracle Database can optimize the performance of table and index scans by performing the scans inside the Exadata Storage Server rather than transporting all the data to DB server. This can also be applied for encrypted and compressed data.

Smart Scan is not governed by the Optimizer but it is influenced by the results of query optimization.

Smart Scan Example:

Specific Requirements for Smart Scan

Smart Scan is only possible for full segment scan like FTS, Index Fast Full Scan etc.
Smart Scan can happen only for SQL statements having atleast one single-table(filter) predicate.
Smart Scan can only be used with direct-path reads. Hence parameter _serial_direct_read should be set TRUE (session/system) for smart scan to happen.
Smart Scan must be enabled within the database. The parameter CELL_OFFLOAD_PROCESSING controls the smart scan and should be set TRUE (default).

To enable Smart Scan for a particular SQL statement, OPT_PARAM hint can be used as follows.
SELECT /*+ OPT_PARAM('cell_offload_processing' 'true') */ ...;

Situations Preventing Smart Scan

1. Scan on a clustered table
2. Scan on an index-organized table
3. Fast full scan on a compressed index
4. Fast full scan on a reverse key indexes
5. The table has row-level dependency tracking enabled
6. The ORA_ROWSCN pseudo column is being fetched and so on.
NOTE: Having STORAGE keyword in the plan for FTS or Index scan does not guarantee the Smart Scan always. Some case the smart scan does not even happen though the plan shows. One such example is when there is high CPU usage on the storage cells, then smart scan does not happen and the storage server return the additional blocks to DB server to process the request rather than processing them inside the cell.

Troubleshooting Smart Scan

Example: Following query can be used to get the metric statistics about smart scan.

### First execute the problem SQL and then run the below script in same session ###

SQL> SELECT a.name, b.value/1024/1024 MB FROM v$sysstat a , v$mystat b WHERE
a.statistic#=b.statistic# AND
(a.name = 'physical read total bytes' OR
a.name = 'physical write total bytes' OR
a.name = 'cell physical IO interconnect bytes' OR
a.name = 'cell physical IO bytes eligible for predicate offload' OR
a.name = 'cell physical IO bytes saved during optimized file creation' OR
a.name = 'cell physical IO bytes saved during optimized RMAN file restore'
OR a.name = 'cell IO uncompressed bytes' OR
a.name = 'cell physical IO interconnect bytes returned by smart scan' OR
a.name = 'cell physical IO bytes saved by storage index');

NAME                                                                     MB
---------------------------------------------------------------- ----------
physical read total bytes                                          5372.53
physical write total bytes                                               0
cell physical IO interconnect bytes                                472.12  <===
cell physical IO bytes saved during optimized file creation              0
cell physical IO bytes saved during optimized RMAN file restore          0
cell physical IO bytes eligible for predicate offload              5372.53
cell physical IO bytes saved by storage index                            0
cell physical IO interconnect bytes returned by smart scan         472.12  <===
cell IO uncompressed bytes                                         5372.53

In the above example output, Smart Scan reduced the amount of I/O transported to DB server by more than 90% (472 MB out of 5373 MB) compared with what would have been required using non-Exadata storage which has to process all 5373 MB. Note that amount of data returned by smart scan (472 MB) matches the amount of data transported across storage interconnect (cell physical IO interconnect bytes). This is an evidence that all I/O is associated with smart scan.

Important CELL Wait events

cell single block physical read : This wait event is equivalent to db file sequential read for a cell. The P1, P2, and P3 columns in the V$SESSION_WAIT for this event to identify the cell.
cell multiblock physical read   : This wait event is equivalent to db file scattered read for a cell. The P1, P2, and P3 columns in the V$SESSION_WAIT for this event to identify the cell.
cell smart table scan              : This wait event appears when the database is waiting for table scans to complete on a cell.
cell smart index scan             : This wait event appears when the database is waiting for index fast full scans.

Storage Indexes

Storage index is a memory-based structure that reduces the amount of physical I/O performed in an Exadata cell. The storage index keeps track of minimum and maximum column values in region index and this information is used to avoid useless I/Os for subsequent SQL execution. This is NOT like database B-tree or Bitmap indexes rather it is a Exadata Storage Server feature. Since the storage index is kept in memory, it disappears when the cell is rebooted.

Refer the following document and check for storage indexes for more details.

Document 1927934.1 Exadata Smart Scan FAQ

ExaWatcher

ExaWatcher replaces OSWatcher in Exadata software versions 11.2.3.3.0 onwards. ExaWatcher resides on both the Exadata database servers (or compute nodes) and storage cell servers.
The ExaWatcher utility is started automatically during boot time. ExaWatcher output can be collected to check for any resources problem. Refer the following document for more details.

Document 1617454.1 ExaWatcher utility on Exadata database servers and storage cells

Exachk

Exachk is similar to ORAchk in Exadata environment. It includes configuration checks for Database Servers, Storage Servers and other components.
Exachk report can be collected to identify any anomalies detected in DB / Cell nodes.

Performance Best Practices

Document 1274475.1 Oracle Exadata Database Machine Performance Best Practices
Document 757552.1 Oracle Exadata Best Practices
  • Use Flash Memory (FDD) available to enable Exadata Smart Flash Cache and Smart Flash Log features.
  • Queries that require indexes on non-Exadata might perform better using Exadata Smart Scan. Consider removing indexes where Smart Scan delivers acceptable performance. Effect of removing unwanted indexes can be tested by making them invisible.
    SQL> ALTER INDEX <NAME> INVISIBLE;
  • Gather Exadata specific system statistics for databases running date warehouse workload. This sets MBRC correctly for Exadata enabling optimizer to do costing more accurately for full table scan thereby favoring smart scans. Do not use WORKLOAD method in Exadata. For OLTP or mixed workload, please do thorough testing of application post Exadata mode system statistics as index scans may not be favored. If you are not in a position to test the effect of using EXADATA system statistics, then stick to the defaults on this platform.
    SQL> exec dbms_stats.gather_system_stats('EXADATA');
  • Exadata Hybrid Columnar Compression provides the choice of compression modes to achieve the optimal disk usage and CPU.
    Document 1910687.1 Exadata Hybrid Columnar Compression (EHCC)
