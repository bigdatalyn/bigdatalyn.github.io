Created from SR 3-21424896343[This section is not visible to customers.]

Applies to:

Oracle Exadata Storage Server Software - Version 11.1.3.1.0 to 19.3.1.0.0 [Release 11.1 to 12.2]
Information in this document applies to any platform.

Goal

 The goal of this document is to share tips on how to optimize your SQLs for the best smart scan performance on Exadata particularly for large reads.

Solution

For maximum smart scan throughput on Exadata you can optimize your SQLs and tune the database for highest performance of smart scans.

Tuning SQLs to Maximize Smart Scan Throughput

Standard SQL Tuning Rules Apply

First of all, most of the normal query turning rules apply to Exadata.  For more on that see:

  Document 1516494.2 - Information Center: SQL Query Performance

And for any SQL performance issue it is helpful to get a SQL Monitor report and possibly SQLT or SQLHC output.

Exadata Smart Scan SQL Best Practices

Specifically for Exadata, we can maximize performance by:

  • Avoiding case insensitive queries.
  • Avoiding REGEXP operators like REGEXP_LIKE.
  • Avoiding single block reads.
  • Choose the best IORM plan and settings to avoid queuing.
  • Utilizing PARALLEL_SERVERS_TARGET

Case insensitive and REGEXP queries often unnecessarily put additional work on the storage cells when the query is offloaded. Optimizing these queries to avoid case insensitivity and REGEXP operators would reduce workload on the storage cells and increase throughput.

Single block reads do not utilize the cells as well as they could be for smart scans. They also increase the IOPs going to the cell disks which may trigger IORM queuing. Note: single block reads are fine for OLTP type queries

Data Collection for Troubleshooting Smart Scan SQL Issues

The steps below are only for smart scan specific issues (i.e. not optimizer plan regressions, etc.)

Collect SQL*Monitor report

-- To collect in-memory SQL monitor report
set pages 0 lines 32767 trimspool on trim on long 1000000 longchunksize 1000000
spool sqlmon.html -- or any other descriptive file name
select dbms_sqltune.report_sql_monitor(sql_id=>'&sql_id',type=>'active') from dual;
spool off;

Are there scenarios when the query performs well? If so, collect SQL*Monitor for the time it worked well (for comparison purposes). If it performed well in the past, you can potentially extract it from the repository using the following (but you'd need to determine the snapshot range when it performed well)

-- get one sql monitor report captured between a range of snapshots
select dbms_auto_report.report_repository_detail(r.report_id, type=>'active')
from dba_hist_reports
where dbid = (select dbid from v$database)
and snap_id between &bsnap and &esnap
and key1 = '&sql_id'
and rownum = 1;

In most cases, once the row source with the smart scan has been identified from SQL*Monitor, the testcase can often be simplified to execute that one row source by selecting rows from the table and adding the predicates (as shown below). Having a simple query that reproduces the problem can often be very useful.

select * from <table> wehre <predicate>

-- if query returns a lot of rows, can use count(*), but that may have other optimizations that kick in
select count(*) from <table> where <predicate>

-- or, you can also simply strip the output using the following. Be careful when using this, as we may not get the true row count from SQL*Monitor.
select /*+ tracing(strip 1) */ * from <table> where <predicate>

In some cases, it may be helpful to collect an AWR report as well - esp if the issues affect multiple SQL statements.

Diagnosing smart scan issues

If you are able to capture a SQL Monitor report for your SQL there are is some good information on smart scans available. In the plan data, find the longest operation and click here:

After you click on the binoculars, we have this data available:

In the example above if you have a simple predicate and are not seeing SI bytes saved then go to the "Reduced Storage Index Savings" section. If you see that passthru is a large percentage of eligabe bytes then go to the "Increased Passthru Bytes" section of the note.

If you are able to gather SQL*Monitor reports for the good vs bad execution:

  • Compare the SQL Monitor reports, verify that the execution plans are the same. If the execution plans are different, then this may be a plan regression. Try and get the same plan again for comparison purposes before proceeding through smart scan diagnosis and follow normal SQL tuning rules.
  • Compare the SQL*Monitor reports, verify the data volumes are comparable. If the execution plans are showing a significant difference in rows or IOs, then verify that the changes in performance are not due to data volume changes.
  • Once you've validated that the plans are identical, and the data volumes are comparable, then with SQL*Monitor:
    • determine where query time is spent (wait events, and row sources), verify that the time is in 'cell smart table scan'
    • verify that the row source for the smart scans (usually TABLE ACCESS STORAGE FULL) have regressed. If the time is not being spent on the smart scan row source, then go back to regular SQL tuning.
  • For the TABLE ACCESS STORAGE FULL row source
    • Check the wait events - are they always 'cell smart table scan' (or cell smart index scan), or did they switch to 'cell multiblock physical read', 'direct path read' or perhaps 'cell single block physical read'?
    • Click on the binoculars to compare possible causes of the regression. This may include lack of SI savings, lack of CC savings, or more passthru bytes, or lack of smart scans (either seeing 'cell multiblock physical read' or 'direct path read' in the wait events).  There are image examples here.
  • In general, for smart scan issues, we would like to see v$cell_state and session statistics as show below:
    • v$cell_state - either save it in CTAS and export the resulting tables, or query v$cell_state before/after query execution

-- save cell state:
create table cell_state_before as select * from v$cell_state;
-- OR
select cell_name, statistics_type, xmltype(statistics_value).getclobval(2,2)
from v$cell_state
where statistics_type = 'PREDIO';

<run query>

-- save cell state
create table cell_state_after as select * from v$cell_state;
-- OR
select cell_name, statistics_type, xmltype(statistics_value).getclobval(2,2)
from v$cell_state
where statistics_type = 'PREDIO';

--
-- export cell_state_before and cell_state_after using exp or expdp, if saving data to a table rather than querying

  • session statistics. If the regression can be reproduced in a single session, then session stats can be collected. Either
    • login and run the query and collect session stats (so session stats only include information related to the query execution), or
    • collect session stats before/after the query(so a comparison can be made)
set lines 128
col name format a60 trunc
col value format 999,999,999,999,999
select n.name, s.value
from v$sesstat s, v$statname n
where n.statistic# = s.statistic#
and s.sid = userenv('SID')
and s.value > 0
and (n.name like 'cell%'
or n.name like 'physical%'
or n.name like 'db%'
or n.name like 'Session%' -- future stat for IORM
or n.name in ('session logical reads','logical read bytes from cache'))
order by n.name;

In some cases, it may be useful to collect the AWR report as well, esp if the issues affect multiple SQL statements.

Below we discuss various reasons smart scan performance could be impacted:

No Smart Scans (cell multiblock physical read)

Direct reads are a prerequisite for performing smart scans. The upper layers determine whether or not a direct read can be done. If the query is not performing a direct read for the row source, using NSMTIO tracing can help determine the root cause. For example, a small segment may not qualify for a direct read.  For more information on NSMTIO tracing see:

Document 2310422.1 - SRDC - Exadata: Smart Scan Not Working Issues

Reduced Storage Index Savings

Reduced storage index savings means more real IOs are performed on the storage cells. This could be because SI hasn't been built yet - in which case subsequent executions of the query would build the storage index.

You can attempt forcing a storage index to get built by running the simple query using a < predicate (e.g. select count(*) from table where column < 0), and then attempt to rerun the test query. If that does not resolve the issue, then determine why SI is not getting built, or not getting used by looking at the output of v$cell_state.  In addition you can monitor storage index usage in SQL Monitor or in v$sysstat / v$sesstat with "cell physical IO bytes saved by storage index".  This statistic should also be in AWR along with a Smart IO section to break it down further.

If the object is new and you are not seeing any storage index usage it could be because the MIN_ACT_SCN has not advanced enough.  For more information on that see:

  Document 2081483.1 - EXADATA and SuperCluster : Check if long running transactions are preventing min active scn from progressing, resulting in Storage Indexes not being used

If you the object is not new and there are no long running transactions you can troubleshoot further with:

  Document 2310422.1 - SRDC - Exadata: Smart Scan Not Working Issues

Reduced Columnar Cache Reasons

Reduced columnar cache savings means more IOs are performed on the storage cells. This could be because the columnar cache has not been built yet - check if there are outstanding population jobs (from AWR or from v$cell_state).

There are a couple of scripts that summarize v$cell_state information for CC available in the OSS vob (oss/utl/ccstats.sql, or a mini-AWR that collects cell_state information from oss/utl/spcell_snap.sql).

Increased Passthru Bytes

Cellsrv can go through a reverse offload when there is high CPU usage on the cells (90-100% CPU). It can also go through passthru for many reasons including database timezone upgrade, low cell server memory (usually < 2-3 GB of free memory), bad SQL causing the offload server to be busy , etc.

If reverse offload is occurring due to high cell CPU, check the predicates being used to see if there are ways to use less CPU for evaluating predicates (as stated above, case-insensitive searches can consume more CPU).  Also check if the workload has changed causing more CPU to be consumed on the cells (for example this would  include possible changes to other databases running on the same cells). Reverse offloading by itself is not always bad, and is merely an indication that the cells are CPU bound, so that data gets sent back to the database for processing instead. The goal is to reduce cell CPU usage.

If passthru is occuring for other reasons - check database statistics for 'cell num smart IO sessions using passthru mode%' .This is also shown in the Exadata sections of the AWR report under "Passthru Reasons"

  • cell num smart IO sessions using passthru mode due to user - the user has explicitly disabled smart scans from occurring
  • cell num smart IO sessions using passthru mode due to timezone - db timezone upgrade disabled smart scans
  • cell num smart IO sessions using passthru mode due to cellsrv - cellsrv, for some reason, decided to perform passthru

If passthru has occurred due to cellsrv, then check v$cell_state for potential other reasons

set lines 120
set pages 100
select cell_name, statistics_type, xmltype(statistics_value).getclobval(2,2)
from v$cell_state where statistics_type='PREDIO';

Use of direct path reads

The wait event can show 'direct path reads' rather than 'cell smart table scan' if there are other resources on the cells that are constrained, preventing the cells from performing smart scans. This would include

  • lack of memory buffers in the cells.
    • Use ExaWatcher to determine if there are a high number of "Number of allocation failures in 1MB pool"
    • Use ExaWatcher and check "Number of active smart IO sessions" and "High water mark of smart IO sessions". If the smart IO sessions are approaching or exceeding 6000, there may be too many concurrent PQ slaves accessing the cells. Determine if the number of concurrent PQ slaves can be reduced by modifying parallel degree, and/or adjusting parallel_server_target.

Check for IORM Queuing

You should also choose an IORM plan / option that fits your workload.  Ideally you should see little queuing in IORM.  To monitor queuing you can look in the "Exadata Top Database Consumers" section of an AWR report search for "Queue Time". If that number is above 10ms you may be suffering from excessive queuing time.  Alternatively you can use the following note to review queuing times on the cell side:

  Document 1337265.1 - Tool for Gathering I/O Resource Manager Metrics: metric_iorm.pl

A quick DCLI command to look for queuing in the last hour would be:

dcli -g ~/cell_group -l root './metric_iorm.pl "where ageInMinutes < 60" | grep "qtime=[1-9][0-9]"'

Tuning PARALLEL_SERVERS_TARGET

To maximize throughput you should also utilize the PARALLEL_SERVERS_TARGET parameter. Normally you can use the documentation guidelines to set PARALLEL_SERVERS_TARGET but in a massively parallel environment we can limit that number to ensure that we are mostly doing smart scans and not direct reads. If the calculation from the documentation exceeds 5000 PQ slaves across all instances then it might be best to set PARALLEL_SERVERS_TARGET anywhere from 5000/instances up to 14000/instances and 50%-80% of PARALLEL_MAX_SERVERS depending on workload. So for example with 4 compute nodes a target range might be 1250-3500 per instance. Ideally we want no more than 4800-5000 active smart scans per cell.  The goal is to minimize direct path reads which can occur if there are too many smart scan clients or if CPU utilization gets too high on the cell servers. If we set PARALLEL_SERVERS_TARGET too low we can have too much queuing(resmgr:pq queued wait event) but if we set it too high we have too many direct path reads. When tuning this parameter it is important to monitor cell CPU, queuing, and direct path read waits. We want the cells to be busy but not overwhelmed.  Here is a table to illustrate:

  PARALLEL_SERVERS_TARGET TOO LOW PARALLEL_SERVERS_TARGET SET CORRECTLY PARALLEL_SERVERS_TARGET TOO HIGH
QUEUING A lot of queuing happening (resmgr:pq queued) wait event. Little or reasonable queuing happening. Zero or nearly zero queuing.
DIRECT PATH READS Almost zero direct path reads. Few direct path reads. Excessive direct path read waits.
CELL CPU Cell CPU is under utilized. Cells are busy but not overwhelmed. Cells are very busy staying at 90-100% CPU.

Identifying Problematic SQLs

If the regressions are not specific to a SQL statement, and there's a perceived regression of general smart scan issues, then ASH can be used to determine SQL statements that waiting on smart scans.

To find SQL statements consuming the most DB time on smart scan waits over the past 5 minutes:

select *
from (
select sql_id, count(*)
from gv$active_session_history
where event = 'cell smart table scan'
and wait_time = 0
and sample_time > sysdate - 5/1440
group by sql_id
order by count(*) desc)
where rownum < 21;

Similarly, to see the top SQL statements consuming the most DB time on smart scan waits over the past 7+ days, a similar query can be executed against dba_hist_active_sess_history instead:

select *
from (
select sql_id, count(*)
from dba_hist_active_sess_history
where event = 'cell smart table scan'
and wait_time = 0
and dbid = &dbid
and snap_id between &bsnap and &esnap
group by sql_id
order by count(*) desc)
where rownum < 21;

Once there are candidate SQL statements, then retrieve the SQL*Monitor reports for the SQL statements, and follow the steps above.

Note that you can also change the wait events in the above queries.  Example: 'cell single block physical read', etc.

AWR report

The AWR report has sections that can help with smart scan diagnosis. Specifically:

  • wait events - indicate the number of waits and the average wait time. The average wait time is a general idea, but each individual wait can vary depending on how much work is offloaded to the storage cells.
  • sysstat (Instance Activity Statistics) - includes a number of useful statistics related to smart scans, such as
    • cell physical IO bytes eligible for predicate offload - # of bytes on storage that are eligible for predicate offload.
    • cell physical IO bytes eligible for smart IO - # of bytes eligible for smart scans, which has the actual bytes that have to be scanned. This can be larger than bytes eligible for predicate offload if the table is in EHCC format.
  • IOStat by Function - this shows total bytes for smart scans. This also shows other types of IO on the system
  • Exadata Smart IO - this has general information regarding smart scans, such as
    • eligible bytes for smart scans
    • bytes saved by SI
    • bytes read from Flash Cache
    • reverse offload bytes
    • passthru bytes

In addition, the Exadata Smart IO sections will also show if there is some imbalance in requests across the cells.

  • Passthru Reasons - if passthru is occurring, this will extract the sysstat for sessions in passthru mode, and will indicate it passthru is due to db timezone upgrades, user settings or cellsrv.
  • Columnar Cache - if the issues are CC related, there are a number of sections related to Columnar Cache including
    • read requests/read bytes from Columnar Cache
    • Columnar Cache efficiency
    • breakdown of Columnar Cache between No Memcompress (CC1), IM Query or IM Capacity (CC2 formats).
    • also shows if there are outstanding population jobs

Additional/Advanced diagnostics

If the steps above do not yield any useful information, it may be necessary to start drilling down into more details as to how a SQL statement is performing. The following are some examples on how ASH can be further used.

To track the performance of a SQL statement, or to see there is a suspicion that certain cells are slow, you can use the following to identify a slow cell:

select sample_time,
p1,
count(*)
from gv$active_session_history
where sql_id = '&sql_id' -- sql_id of test
and event = 'cell smart table scan'
and wait_time = 0
group by sample_time, p1
order by sample_time, count(*) desc;

-- if you see a large number of PQ slaves with the same p1 at the same sample_time, then this may be an indication that the cell needs to be further investigated.
-- To identify the cell:
col cellname format a30
col cell format a30
select cellname,
extractvalue(xmltype(confval),'//cell/name') cell
from v$cell_config_info
where conftype = 'CELL'
and cellhash = &p1;

To get an idea on whether or not some of the waits are "long", you can query the time_waited column. But be wary of performing any aggregations on the time_waited column, and using those aggregations to make conclusions.

select sample_time, time_waited
from gv$active_session_history
where event = 'cell smart table scan'
and wait_time = 0
and sql_id = '&sql_id'
order by sample_time;

If you really want to estimate the average latency of the waits for the SQL statements consuming the most DB time on smart scan waits (over the past 5 minutes), this can be estimated using the following:

-- assumes in memory ASH samples are every second
select *
from (
select sql_id,
estd_db_time_ms,
estd_db_time_ms/estd_waits estd_latency
from (
select sql_id,
-- if the time_waited is > sampling_interval, then count it as a single wait
-- otherwise, the estimated waits = sampling_interval / time_waited
sum(case when time_waited >= 1000000
then 1
else 1000000/decode(time_waited,0,null,time_waited)
end) estd_waits,
-- db time is # of samples * sampling_interval
count(*) * 1000 estd_db_time_ms
from gv$active_session_history
where sample_time > sysdate - 5/1440
and event='cell smart table scan'
and wait_time = 0
group by sql_id)
order by estd_db_time_ms desc)
where rownum < 21;

Similarly, to estimate the average latency of the sql statements consuming the most DB time on smart scan waits over the past 7+ days:

-- assumes on-disk ASH samples are every 10 seconds
select *
from (
select sql_id,
estd_db_time_ms,
estd_db_time_ms/estd_waits estd_latency
from (
select sql_id,
-- if the time_waited is > sampling_interval, then count it as a single wait
-- otherwise, the estimated waits = sampling_interval / time_waited
sum(case when time_waited >= 10000000
then 1
else 10000000/decode(time_waited,0,null,time_waited)
end) estd_waits,
-- db time is # of samples * sampling_interval
count(*) * 10000 estd_db_time_ms
from dba_hist_active_sess_history
where dbid = :dbid
and sample_time > trunc(sysdate) - 7
and event='cell smart table scan'
and wait_time = 0
group by sql_id)
order by estd_db_time_ms desc)
where rownum < 21;

To find cpu intensive queries, you can look at AWR, or you can query v$sqlstats.

-- you can query regular executions, cpu_time, elapsed_time columns, or the delta columns.
-- the delta columns are deltas since the last AWR snapshot. This gives it a 'recency' bias.
select sql_id,
delta_execution_count, delta_elapsed_time, delta_cpu_time, delta_user_io_wait_time
delta_physical_read_requests, delta_physical_read_bytes,
delta_io_interconnect_bytes, delta_cell_offload_elig_bytes
from v$sqlstats
where cpu_time > &threshold
order by delta_cpu_time desc;
