Abstract

This article investigates the behaviour of the In-Database Row Archiving feature introduced in Oracle Database 12c.

Background

The In-Database Row Archiving feature allows rows to be archived by marking the rows as inactive. Inactive rows remain in the table but are no longer visible to the application by default.

The feature can be enabled using ALTER TABLE ... ROW ARCHIVAL and disabled using ALTER TABLE ... NO ROW ARCHIVAL When the feature is enabled for a table:

  1. an invisible hidden column ORA_ARCHIVE_STATE is added to the table
  2. SQL statements are modified transparently to include additional predicates to filter inactive rows.

Although marking a row inactive or active requires explicit manipulation of the value of the ORA_ARCHIVE_STATE column, the feature is intended to be transparent to existing application code.

Annotated Examples

In the following examples, the In-Database Row Archiving feature is enabled for table t1, not for other tables. Table t1 contains one active row, and one inactive row.

Left Outer Join

When t1 is on the left of an outer join, the query returns both active and inactive rows, when only active rows should be returned.

Possible workarounds include:

  • using Oracle syntax rather than ANSI syntax
  • alter session set "_optimizer_ansi_rearchitecture" = false
  • projecting ORA_ARCHIVE_STATE using an inline view, which defeats the objective of transparency.

See BUG#26099981 and BUG#27355540.

Right Outer Join

When t1 is on the right of an outer join, the query may return a zero (active) value for column ORA_ARCHIVE_STATE depending on the plan, when a NULL value should be returned. The nested loop plan returns the expected result, while the hash join and merge join (not shown here) plans return the unexpected result.

A possible workaround is to project ORA_ARCHIVE_STATE using an inline view, which defeats the objective of transparency..

See BUG#27369598.

Insert / Merge / Create Table As Select

It would normally be possible to copy the rows of a table to a new table using different methods, for example:

  1. CREATE TABLE ... AS SELECT
  2. CREATE TABLE followed by INSERT INTO ... SELECT ...
  3. CREATE TABLE followed by MERGE INTO ... SELECT ... with a NOT MATCHED clause

The second method produces the expected result i.e. only the active rows are copied to the target table.

The first method copies active and inactive rows. No filter predicate is added. Oracle By Example indicates that this is expected behaviour but if so, it seems inconsistent.

The third method copies active and inactive rows. No filter predicate is added.

Create Table As Select

It is not possible to include ROW ARCHIVAL or NO ROW ARCHIVAL in a CREATE TABLE ... AS SELECT ... statement. It is possible to include ROW ARCHIVAL in a CREATE TABLE statement. This seems inconsistent with other features, for example COMPRESS and NOCOMPRESS. 

create table t1(m not null, n not null, constraint pk_t1 primary key (m)) row archival as select 1, 1 from dual
*
ERROR at line 1:
ORA-01722: invalid number


create table t1(m not null, n not null, constraint pk_t1 primary key (m)) no row archival as select 1, 1 from dual
                                                                                                              *
ERROR at line 1:
ORA-38397: table is not enabled for the ILM feature

Test Case 

whenever sqlerror continue
drop table t0 purge;
drop table t1 purge;
drop table tc purge;
drop table ti purge;
drop table tm purge;
whenever sqlerror exit failure rollback

column ora_archive_state format a17
set echo on feedback 1024 linesize 132 pagesize 66 trimspool on

create table t0(m not null, n not null, constraint pk_t0 primary key (m)) as select 0, 0 from dual;
create table ti(m not null, n not null, constraint pk_ti primary key (m)) as select 0, 0 from dual where 1 = 0;
create table tm(m not null, n not null, constraint pk_tm primary key (m)) as select 0, 0 from dual where 1 = 0;

/*
the following CTAS fails, so use two steps rather than one ...
create table t1(m not null, n not null, constraint pk_t1 primary key (m)) row archival as select 1, 1 from dual;
create table t1(m not null, n not null, constraint pk_t1 primary key (m)) row archival as select 1, 1 from dual
*
ERROR at line 1:
ORA-01722: invalid number

for the record, the following CTAS fails too
create table t1(m not null, n not null, constraint pk_t1 primary key (m)) no row archival as select 1, 1 from dual;
create table t1(m not null, n not null, constraint pk_t1 primary key (m)) no row archival as select 1, 1 from dual
                                                                                                              *
ERROR at line 1:
ORA-38397: table is not enabled for the ILM feature
  */

create table t1(m not null, n not null, constraint pk_t1 primary key (m)) as select 1, 1 from dual;
alter table t1 row archival;
insert into t1(m, n, ora_archive_state) values (2, 1, 1);
exec dbms_stats.gather_table_stats(USER, 'T1')

REM single table query
select m, n from t1 order by m;
select * from table(dbms_xplan.display_cursor);

REM outer join with ILM table on the left
select t1.m, t1.m
from t1 left outer join t0 on t1.m = t0.m order by t1.m;
select * from table(dbms_xplan.display_cursor);

REM outer join with ILM table on the right
select /*+ LEADING(t0) INDEX(t0) INDEX(t1) USE_NL(t1) */ t0.m, t1.m, t1.m, t1.ora_archive_state
from t1 right outer join t0 on t1.m = t0.m order by t0.m;
select * from table(dbms_xplan.display_cursor);
select /*+ LEADING(t0) FULL(t0) FULL(t1) USE_HASH(t1) */ t0.m, t1.m, t1.m, t1.ora_archive_state
from t1 right outer join t0 on t1.m = t0.m order by t0.m;
select * from table(dbms_xplan.display_cursor);

select * from table(dbms_xplan.display_cursor);

REM CTAS
create table tc as select * from t1;
select * from table(dbms_xplan.display_cursor);
select m, n from tc order by m;

REM insert
insert into ti select * from t1;
select * from table(dbms_xplan.display_cursor);
select m, n from ti order by m;

REM merge
merge into tm dst using (select * from t1) src
on (dst.m = src.m)
when not matched then insert values (src.m, src.n);
select * from table(dbms_xplan.display_cursor);
select m, n from tm order by m;

exit
