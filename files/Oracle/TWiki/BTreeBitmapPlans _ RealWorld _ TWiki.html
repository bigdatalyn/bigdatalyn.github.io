Combining B-Tree Indexes

  • Abstract
  • Examples
    • B-Tree Indexes
    • B-Tree and Bitmap Indexes
  • Test Case

Abstract

This article describes different methods of combining information from B-Tree indexes at runtime. These methods provide new access paths that can be more efficient than using just one of the indexes albeit less efficient than an appropriate multi-column B-Tree index or equivalent bitmap indexes.

Examples

The examples use a table with two columns m and n, each having 1000 unique values, producing 1000000 rows in total. .

B-Tree Indexes

In the following examples, there is a single column B-Tree index on column m and on column n.

In the first query, the ROWIDs from the INDEX RANGE SCAN operations are converted to bitmaps, and the bitmaps are combined using BITMAP AND.

The second query demonstrates that it is also possible for a suitable query to use BITMAP OR.

The third query returns column values rather than a count. The bitmap result from BITMAP AND is converted to ROWIDs using a BITMAP CONVERSION TO ROWIDS operation before retrieving rows by ROWID in the usual way using TABLE ACCESS BY INDEX ROWID [BATCHED]. 

SQL> create index jz_m on jz(m);

Index created.

SQL> create index jz_n on jz(n);

Index created.

SQL> set autotrace traceonly explain
SQL> select count(*) from jz where m = 123 and n = 321;

B-Tree and Bitmap Indexes

In the following examples, there is a single column B-Tree index on column n- and the B-Tree index on column =m is replaced by a bitmap index on column m.

In the first query, the ROWIDs from the INDEX RANGE SCAN operation are converted to bitmaps, and the bitmaps are combined using BITMAP AND.

It is commonplace for many complex applications to specify _b_tree_bitmap_plans=false at database-level. The parameter is specified in the second query but it is ineffective. Why?

When there is one or more bitmap indexes on the table, Oracle will continue to consider plans involving bitmaps created from index range scans of B-Tree indexes. The parameter is effective for tables with only B-Tree indexes:

  • From the description of the parameter: enable the use of bitmap plans for tables w. only B-tree indexes
  • Not-a-bug BUG#4309271

Index dropped.

SQL> create bitmap index jz_m on jz(m);

Index created.

Test Case 

drop table jz purge;
create table jz as
with q as (select rownum r from dual connect by rownum <= 1e3)
select /*+ LEADING(q1) USE_NL(q2) */ q1.r as m, q2.r as n, cast('filler' as char(64)) as filler from q q1 cross join q q2;
REM select * from table(dbms_xplan.display_cursor);
set echo on
create index jz_m on jz(m);
create index jz_n on jz(n);
set autotrace traceonly explain
select count(*) from jz where m = 123 and n = 321;
select count(*) from jz where m = 123 or n = 321;
select * from jz where m = 123 and n = 321;
select /*+ OPT_PARAM('_b_tree_bitmap_plans' 'false') */ count(*) from jz where m = 123 and n = 321;
drop index jz_m;
create bitmap index jz_m on jz(m);
select count(*) from jz where m = 123 and n = 321;
select /*+ OPT_PARAM('_b_tree_bitmap_plans' 'false') */ count(*) from jz where m = 123 and n = 321;
set autotrace off echo off
