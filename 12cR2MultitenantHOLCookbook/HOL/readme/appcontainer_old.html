What are Application Containers?

An application container consists of an application root and all of the application PDBs associated with it. The Application root is a special PDB, which contains the data model definition along with any metadata and common data for any application. Application Containers enables you to create a common application data model that can be shared across multiple tenants participating in the federation. You can also create and maintain a common data source that can be referenced by individual tenants.

An application root enables you to create application common objects that are shared only by the application root and the application PDBs that belong to the application root. Application common objects are not visible to the CDB root, other application roots, or PDBs that do not belong to the application root. An application root cannot belong to another application root, and a PDB can belong to either no application roots or to only one application root. You can create an application seed for an application container. The application seed is optional, but, if it exists, you can use it to create application PDBs quickly that match the requirements of the application container.

An application seed enables instant provisioning of application PDBs that are created from it.

We have two labs in this section.

Lab 1: Create an application that will be shared by multiple PDBs


The first one we will go through the creation of application root, installing an application followed by setting up two application PDBs (Robots and Dolls).

My Toy Inc. sells two toys Robots and Dolls. Revenue information for these two products are stored in two separate PDBs, however the table definition for this sales application is the same. So we will create an application container, define a metadata linked table which will allow us to share the table definition in both the PDBs. The definition is shared but data in each PDB will be isolated and is not accessible from another PDB.At the end we will execute a query that will aggregate data from the two applciation PDBs using a containers clause.


  • Open a terminal window
  • change directory to /u01/HOL/app_containers/
  • source the environment. At command prompt execute $source cdb1.env
  • sqlplus /nolog
  • At the SQL prompt execute SQL>@01.app_container_lab.sql


    • This is a self guided script that will walk you through the basics of Application containers.

Lab 2: Logical partitioning of data in pluggable database using container maps


Container maps enable the partitioning of data at the PDB level when the data is not physically partitioned at the table level.


In this exercise, we will use the well known EMP and DEPT table. Imagine in every department of our company there are hundreds of thousands of employees. We have a need to separate this data into few pluggable databases but we cannot make any changes to the application i.e. existing queries should continue to work as before. Since deptno column is used by most of our application queries we will use that as the container map column.


  • Open a terminal window
  • change directory to /u01/HOL/app_containers/
  • source the environment. At command prompt execute $source cdb2.env
  • sqlplus /nolog
  • At the SQL prompt execute SQL>@02.container_map_lab.sql
